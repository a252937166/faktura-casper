# The x402 risk oracle — settlement modes

Every underwriting run produces a verified risk report. Other agents buy it
machine-to-machine over **HTTP 402 Payment Required** with on-chain CSPR
settlement: `GET /api/risk/:id` answers `402` + x402 `PaymentRequirements`
(scheme `exact`, network `casper:casper-test`), the buyer settles on-chain,
then replays the request with proof headers and gets the report.

The report is worth paying for because it is **auditable**: it carries the
`decisionHash` that the underwriter anchored on-chain at decision time, so the
buyer can verify the oracle isn't rewriting history (see the
`verify_decision_hash` MCP tool).

Two settlement modes, selected with `X402_MODE`:

## `native-demo` (default — what the hosted demo and e2e use)

Facilitator-less, fully self-contained:

1. `GET /api/risk/7` → `402` with `payTo` (the oracle's key), `maxAmountRequired`
   (2.5 CSPR) and a one-time `transferIdNonce`.
2. Buyer sends a **native CSPR transfer** to `payTo` carrying the nonce as the
   Casper `transfer id` (the demo buyer, `npm run x402-demo`, signs it
   in-process with casper-js-sdk).
3. Buyer retries with `PAYMENT-SIGNATURE: <deploy-hash>` and
   `PAYMENT-NONCE: <nonce>`.
4. The oracle verifies the deploy over RPC — executed, right payee, amount ≥
   price, transfer id == nonce, not replayed — and releases the report.

Trade-off: settlement is a plain transfer rather than a token authorization,
so it needs no CEP-18 token, no facilitator service and no extra approvals —
right-sized for a self-contained testnet demo while keeping the x402 wire
format (402 status, `PaymentRequirements`, proof header, per-charge nonce).

## `official-facilitator` (integration point)

For deployments that use a standard x402 **facilitator** service
(e.g. a CEP-18 `transfer_with_authorization` flow):

```bash
X402_MODE=official-facilitator X402_FACILITATOR_URL=https://<facilitator> npm start
```

The oracle keeps issuing the same `402` challenges but delegates proof
verification to the facilitator's `POST /verify` API and trusts its verdict.
This is wire-compatible with facilitator-based x402 clients; it is **not
exercised in the hackathon demo** (no public facilitator runs on
`casper-test`), which is exactly why the default mode exists.
